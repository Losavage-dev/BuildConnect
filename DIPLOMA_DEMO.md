# BuildConnect — сценарий защиты диплома (10–15 минут)

Документ для быстрой подготовки демо: что показать комиссии, в каком порядке, какими аккаунтами.

## Перед защитой (30–60 минут)

1. **Supabase Cloud**: `npx supabase link` → `npx supabase db push` (32 миграции, без ошибок).
2. **Seed** (SQL Editor, по порядку):
   - `supabase/seed_test_accounts.sql` — заказчик, подрядчики, поставщик
   - `supabase/seed_moderator_account.sql` — модератор
3. **Auth**: Site URL и Redirect URLs = URL Vercel (и `http://localhost:8080/**` для репетиции).
4. **Vercel**: env `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`, деплой Preview/Production.
5. Локально: `npm run smoke` и `npm run build` — оба должны пройти.

Подробный чеклист: [DEPLOY_CHECKLIST.md](DEPLOY_CHECKLIST.md).

## Сценарий демонстрации (рекомендуемый порядок)

### 1. Гостевой обзор (1–2 мин)

- Главная `/` — позиционирование B2B для строительства KZ.
- `/catalog` — список компаний, фильтр по городу/категории, вкладка **«Для вас»**.
- `/tenders` — публичные тендеры, блок **«Рекомендуемые тендеры»**, кнопка «Подробнее».

**Демо рекомендаций (1 мин):** войти как `contractor1@test.com` → открыть 2–3 компании в каталоге → главная: блок **«Рекомендуем вам»** → каталог: переключатель **«Для вас»** → тендеры с сортировкой **«Для вас»**.

**Тезис для комиссии:** маркетплейс объединяет поиск исполнителей и закупок в одной платформе.

### 2. Заказчик: тендер и сделка (3–4 мин)

Войти как `client@test.com`.

1. Создать тендер (или открыть существующий из seed).
2. Показать отклик подрядчика (если есть в seed — открыть «Мои тендеры» → отклики).
3. Принять отклик → чат → **«Завершить заявку»**.
4. Оставить отзыв (доступен только после завершения).

**Тезис:** сквозной цикл «публикация → отклик → договорённость в чате → закрытие → репутация».

### 3. Подрядчик: профиль и портфолио (2–3 мин)

Войти как `contractor1@test.com`.

1. Профиль компании — описание, услуги, документы на верификацию (если не verified).
2. Портфолио — проект, редактирование, удаление фото.
3. Отклик на чужой тендер (кратко).

**Тезис:** исполнитель ведёт витрину и подтверждает квалификацию документами.

### 4. Жалоба и модерация (4–5 мин)

1. Под любым аккаунтом (кроме модератора): на карточке компании/тендера — **«Пожаловаться»** → причина + комментарий.
2. Выйти → `moderator@test.com` → **Рабочее место модератора**:
   - Очередь жалоб, деталь жалобы, ссылка на объект.
   - Действия: скрыть, предупреждение владельцу, временный бан, «Закрыть без санкций».
   - Вкладки: **журнал действий**, **архив**.
3. Верификация компаний: одобрить / отозвать «Проверено» / приостановить.

**Тезис:** доверие обеспечивается не только рейтингом, но и ручной модерацией с аудитом (`moderation_actions`, уведомления).

### 5. Техническая часть (1–2 мин, по вопросам)

- **Стек:** React + Vite + TypeScript, Supabase (PostgreSQL, RLS, Auth, Storage, Realtime).
- **Безопасность:** Row Level Security, роли `client` / `contractor` / `supplier` / `moderator` / `admin`.
- **Валидация:** централизованные Zod-схемы перед отправкой в API.
- **Деплой:** фронт Vercel, БД Supabase Cloud; миграции версионируются в `supabase/migrations/`.

## Частые вопросы на защите

| Вопрос | Ответ |
|--------|--------|
| Почему Supabase, а не свой backend? | Быстрый MVP с RLS, realtime и auth; фокус диплома — предметная область и UX |
| Как защищены данные модератора? | Политики RLS + RPC только для роли moderator/admin |
| Что если жалоб много? | Эскалация при 3+ жалобах на один объект, приоритет в очереди |
| Масштабирование? | Индексы, пагинация списков; для production — Edge Functions, CDN |

## Если что-то сломалось на площадке

1. Проверить `.env` на Vercel и Redirect URLs в Supabase Auth.
2. `npm run smoke` — таблицы и миграции.
3. Последняя миграция уведомлений: `20260528120000_notifications_grant_unban_warning.sql`.
4. Повторно выполнить seed только на **пустой** или тестовой БД (скрипты идемпотентны частично — см. комментарии в SQL).
